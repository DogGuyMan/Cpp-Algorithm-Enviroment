# 회전(Rotation) 시간복잡도 고찰 — `std::rotate` vs `deque` 양끝 pop/push

> **질문**: 길이 `n`인 시퀀스를 `k`칸 회전시킬 때, `std::rotate` 한 번 호출과 `deque`의 "앞에서 빼서 뒤에 넣기"를 반복하는 것 중 무엇이 유리한가?

핵심 결론부터: **두 방식은 비교 대상이 다르다.** `std::rotate`는 *"임의의 `k`만큼 한 방에"* 돌리는 `Θ(n)` 연산이고, deque 양끝 이동은 *"한 칸씩"* 돌리는 `Θ(1)` 연산이다. 그래서 **회전량 `k`가 `n`에 비해 작을수록 deque가, `k`가 클수록(또는 한 번에 끝내야 할수록) `std::rotate`가** 유리하다.

---

## 1. `std::rotate` — 한 번에 `Θ(n)`

```cpp
// [first, middle, last) → middle이 새로운 first가 되도록 회전
auto newFirst = std::rotate(v.begin(), v.begin() + k, v.end());
// 반환값: 원래 첫 원소(*first)가 이동한 위치 == first + (last - middle)
```

- **시간복잡도: `Θ(n)`**, `n = std::distance(first, last)`. 표준이 보장하는 건 *"`[first, last)` 거리에 선형"* 이다.
- **중요한 함정**: 비용이 **`k`와 무관**하다. `k = 1`이든 `k = n/2`이든 똑같이 `Θ(n)`이다. 회전을 적게 하든 많이 하든 **모든 원소를 건드린다.**
- **제자리(in-place), 추가 할당 없음**: `O(1)` 추가 공간.

### 대표 구현 3가지 (모두 `Θ(n)`)

| 알고리즘 | 발상 | 비고 |
|---|---|---|
| 3-reversal | `reverse[first,middle)` + `reverse[middle,last)` + `reverse(전체)` | 가장 단순, 캐시 친화적 |
| juggling (cycle leader) | `gcd(n, k)`개 사이클을 따라 한 칸씩 밀기 | 원소 이동 `n - gcd(n,k)`회 |
| gcd 기반 swap_ranges | 양 블록을 맞바꿔 가며 축소 | `<algorithm>` 일부 구현 |

> 정확한 swap/move 횟수는 구현마다 다르지만(표준은 명시 안 함), **점근적으로는 전부 `Θ(n)`**. 연속 메모리(`vector`)에서는 3-reversal이 캐시 지역성이 좋아 상수가 가장 작다.

## 2. `deque` 양끝 pop/push — 한 칸당 `Θ(1)`

```cpp
// 앞 → 뒤로 한 칸 회전 (왼쪽 회전)
int front = dq.front();
dq.pop_front();
dq.push_back(front);

// 뒤 → 앞으로 한 칸 회전 (오른쪽 회전)
int back = dq.back();
dq.pop_back();
dq.push_front(back);
```

- **`push_front` / `push_back` / `pop_front` / `pop_back` 모두 amortized `O(1)`.** `deque`는 고정 크기 블록들을 가리키는 맵(map of blocks) 구조라, 양끝 삽입·삭제가 원소 이동 없이 상수 시간이다.
- 따라서 **한 칸 회전 = `Θ(1)`**. `k`칸 회전은 이걸 `k`번 = **`Θ(k)`**.
- 방향을 골라 짧은 쪽으로 돌면 **`Θ(min(k, n−k))`** 까지 줄일 수 있다.

### `O(1)`의 단서 (amortized인 이유)

- `push_*`가 현재 끝 블록을 다 채우면 **새 블록을 할당**한다(가끔). 아주 드물게 블록 포인터 맵 자체를 재할당한다. 이 비용이 분산되어 **amortized `O(1)`**.
- **할당이 발생할 수 있다** — `std::rotate`(무할당)와 달리, deque 방식은 메모리 트래픽이 있다.
- 캐시 지역성은 `vector`보다 나쁘다(블록 분절). 같은 `Θ(n)`이라도 **상수가 더 크다.**

## 3. 시나리오별 비교 — 여기가 고찰의 핵심

### 시나리오 A: "한 번에 `k`칸 회전"

| 방법 | 시간 | 비고 |
|---|---|---|
| `std::rotate` 1회 | `Θ(n)` | `k`와 무관, 무할당, 연속 메모리면 상수 작음 |
| deque pop/push `k`회 | `Θ(min(k, n−k))` | **`k ≪ n`이면 압도적으로 빠름** |

- `k = 1`: `std::rotate`는 `Θ(n)`, deque는 **`Θ(1)`**. → deque 완승.
- `k ≈ n/2`: 둘 다 `Θ(n)`. → 상수가 작은 `std::rotate`(특히 `vector`)가 실측 우위.

### 시나리오 B: "한 칸씩 단계적으로, 중간 상태를 매번 확인" (라운드로빈, 슬라이딩 등)

`t`번 단계 회전한다고 할 때:

| 방법 | 총 시간 |
|---|---|
| deque 단계 회전 `t`회 | **`Θ(t)`** |
| `std::rotate(…, +1, …)` 를 `t`회 | `Θ(n · t)` |

- `std::rotate`로 한 칸씩 `t`번 돌리면 매번 `Θ(n)` → **`Θ(n·t)`**, `t = n`이면 `Θ(n²)`.
- deque는 단계마다 `Θ(1)` → 총 `Θ(t)`. **단계적 회전은 deque가 정답.**

---

## 4. 정리 표

| 기준 | `std::rotate` | `deque` 양끝 pop/push |
|---|---|---|
| 한 칸 회전 | `Θ(n)` | **`Θ(1)`** |
| `k`칸 회전(1회) | `Θ(n)` (k 무관) | `Θ(min(k, n−k))` |
| 단계 회전 `t`회 | `Θ(n·t)` | **`Θ(t)`** |
| 추가 공간 | `O(1)`, 무할당 | amortized, 가끔 할당 |
| 캐시 지역성 | 좋음(`vector` 연속) | 나쁨(블록 분절) |
| 반복자/참조 무효화 | 값이 이동(위치 의미 변함) | push: 모든 *반복자* 무효화(참조는 유효), pop: 제거 원소만 무효화 |
| 컨테이너 제약 | ForwardIterator면 됨 | deque 전용 동작 |

---

## 5. 선택 가이드

- **임의의 `k`를 한 번에, 큰 `k`, `vector` 위에서** → `std::rotate`. 무할당·캐시 친화·한 줄.
- **회전량이 작거나(`k ≪ n`), 한 칸씩 진행하며 매 단계를 봐야 하거나, 양끝 큐 동작이 빈번** → `deque` 양끝 이동. 단계당 `Θ(1)`.
- **헷갈릴 때 한 줄 기준**: *"전체를 한 번 갈아엎느냐(`rotate`) vs 끝의 한 원소만 옮기느냐(`deque`)"*. 옮길 원소 수가 곧 비용이다.

> **주의 — `deque`에도 `std::rotate`를 쓸 수 있다.** `deque`는 RandomAccessIterator라 `std::rotate(dq.begin(), dq.begin()+k, dq.end())`가 동작하며 이때도 `Θ(n)`이다. 즉 "deque가 빠르다"가 아니라 **"양끝 pop/push라는 단계적 연산이 `Θ(1)`"** 이라는 점이 핵심이다. 컨테이너가 아니라 **연산의 종류**가 복잡도를 가른다.

---

## 부록 — 작은 벤치 직관

`n = 1000`, 전체를 한 바퀴(`n`칸) 회전한다고 하자.

- `std::rotate` 1회: `Θ(n)` = 약 1,000 단위.
- deque 단계 회전 `n`회: `Θ(n)` = 약 1,000 단위 (단, 상수가 더 큼).
- `std::rotate` 한 칸씩 `n`회(❌ 안티패턴): `Θ(n²)` = 약 1,000,000 단위.

→ **"한 방에 끝낼 거면 `std::rotate`, 한 칸씩 갈 거면 `deque`"**, 그리고 **`std::rotate`를 루프 안에서 한 칸씩 돌리는 건 피한다.**
